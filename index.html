01 — Hardware Build

The Platform

Starting from a donated electric wheelchair from the Mary Campbell Center, I designed and wired a complete autonomous perception stack — from 12V lithium power distribution through edge compute to sensor drivers.

NVIDIA Orin Nano — edge inference & ROS2 host
Pico Flexx — 3D depth camera, PMD ToF
RPLidar A1M8 ×2 — dual 2D scanning, 360°
BNO08x IMU — 9-DOF orientation + gyro
HC-SR04 ×4 — ultrasonic near-field ranging
ESP32 — MCU bridge, custom serial protocol
MCP4728 DAC + CD4066 — motor control interface
Power architecture

Two 12V 54Ah Dakota Lithium batteries feed a circuit breaker, then a DC-DC converter steps to 5V for the compute stack. All wiring and electrical housing designed and routed by hand.

02 — Interesting Component

The ESP32 Hardware Bridge
& Custom Serial Protocol

Operating Principle

ESP32 → ROS2 · UART · Custom framed protocol · Gyro-merge trick

The most technically interesting component I built is the ESP32Bridge — a custom hardware abstraction layer that translates raw microcontroller sensor data into properly typed ROS2 messages. Off-the-shelf serial parsers couldn't handle the heterogeneous mix of IMU quaternions, gyro rates, and ultrasonic distances I needed to multiplex over a single UART link.

I designed a lightweight framing protocol using angle-bracket delimiters and prefix tags. Each message type is self-identifying — the parser routes on the prefix string and never needs to track state across frames. This makes the protocol trivially extensible: adding a new sensor type is just a new prefix. 

# Wire format — self-framing, prefix-routed
<IMU_Q, 0.12, 0.45, 0.78, 0.32>  ← orientation quaternion
<IMU_G, 0.042>                    ← gyro rotation rate
<US, 45, 23, 67, 12>             ← 4 ultrasonic distances (cm)

The most elegant problem I solved was the gyro-merge trick. ROS2's sensor_msgs/Imu requires orientation and angular velocity in a single message — but the ESP32 sends them as separate frames to minimize packet size. My node buffers the latest gyro_z value and stamps it onto the next quaternion frame before publishing. Zero data loss, no extra bandwidth. 

if prefix == "IMU_G":
    self._gyro_z = float(parts[1])  # buffer it
elif prefix == "IMU_Q":
    imu_msg.angular_velocity.z = self._gyro_z  # attach
    self.imu_pub.publish(imu_msg)
elif prefix == "US":
    # Each distance → a single 3D point for SLAM
    cloud = pc2.create_cloud_xyz32(
        header, [[dist_m, 0.0, 0.0]]
    )

The ultrasonic-to-PointCloud2 conversion is another choice worth noting: rather than writing a custom message type, I map each distance reading to a single 3D point [dist, 0, 0] and let ROS2's transform system rotate it into the correct world frame. The perception pipeline sees it as just another point cloud — no special casing required.

This component directly demonstrates the kind of low-level sensor integration work WindBorne does — custom drivers, calibration logic, and data pipelines for instruments that need to work reliably in harsh, uncontrolled environments.

03 — Design for Ease of Use

3-Zone Obstacle Avoidance
with Hysteresis

Why hysteresis is the key design choice

Without hysteresis, obstacle avoidance systems oscillate — the wheelchair spins back and forth at a threshold boundary, confusing and alarming users. I solved this with two asymmetric thresholds: avoidance activates at 20 cm but only clears at 35 cm. This 15 cm dead-band means the system commits to an action before reversing it, giving smooth, predictable behavior at the cost of a tiny bit of conservatism.

This is the difference between a system that technically works and one a person with a disability would actually trust. Every design choice in the navigation layer prioritized behavioral predictability over raw performance.

3-zone control logic
ZONE 1 — Clear
Full speed · proportional heading
> 1.2 m
ZONE 2 — Caution
Reduced speed · turn to clearer side
0.55 – 1.2 m
ZONE 3 — Stop
Full stop · rotate in place
< 0.55 m
Hysteresis thresholds
ACTIVATE: 20 cm · CLEAR: 35 cm

Sensor fusion, not sensor priority

Rather than trusting a single sensor, the node subscribes to all five data streams (2× LiDAR, 2× depth camera, 4× ultrasonic) and takes the minimum distance per zone. Any sensor detecting danger triggers the response. This means a sensor dropout doesn't create a blindspot — it just means slightly less coverage.

RViz live visualization

Real-time point cloud visualization in RViz gives operators an immediate, intuitive view of what the robot sees — colored clouds by sensor source, URDF wheelchair model, and ultrasonic positions as 3D markers. Debugging in the field requires visual feedback, not log parsing.

MATLAB simulation-first validation

Before any hardware deployment, I validated the 3-zone logic in a MATLAB simulation with a 3D STL wheelchair model and 42 ray-cast fake LiDAR beams — the same algorithm as the real node. Identical thresholds, identical state machine. Hardware got parameters that were already proven, not guesses.

05 — Why WindBorne

Direct Skill Alignment

WindBorne's sensors internship is about building, calibrating, and understanding instruments that work in the harshest environments on Earth. Here's exactly how this project maps to that.

01
Hardware Drivers from Scratch
→ WB: sensor manufacturing & calibration infrastructure
Wrote hardware drivers for 3 distinct sensor types (ToF depth camera, 2D LiDAR, 9-DOF IMU) — not wrappers, actual driver logic from SDK up. The discipline of making sensors produce trustworthy data is identical to calibration work.
02
Bias, Drift, and Data Integrity
→ WB: investigate bias, systematic error, contamination
The gyro-merge trick exists precisely because IMU data has timing artifacts — I had to understand the sensor's output timing to patch it correctly. Hysteresis thresholds were set by analyzing real detection drift at zone boundaries, not guesswork.
03
Simulation → Hardware Validation
→ WB: test sensors in flight, analyze flight performance data
Built a full MATLAB simulation with identical control logic before touching hardware. The mentality — validate the algorithm before trusting it in the field — is the same discipline that flight-testing meteorological sensors demands.
04
Custom Calibration Infrastructure
→ WB: improve calibration infrastructure
Designed the serial protocol and ROS2 bridge from scratch — this is calibration infrastructure. Choosing the right coordinate frame transforms, sensor origin offsets, and thresholds all require the same iterative measurement and correction mindset.
05
Python + Linux + Data Analysis
→ WB: Python proficiency, Linux/bash, data analysis
Every node is pure Python on Ubuntu. I debug over serial terminals, profile node timing in bash, and parse sensor logs to tune thresholds. The toolchain is identical to what WindBorne's data analysis pipeline requires.
06
End-to-End Ownership
→ WB: high agency, own projects end-to-end
I started with a donated wheelchair and ended with a working autonomous perception system — wiring, firmware, drivers, ROS2 nodes, simulation, and deployment. No handoff. The same ownership WindBorne needs for sensors going into a global balloon constellation.
